PIE Players · Item player overview

The new PIE Item Player

A quick orientation to @pie-players/pie-item-player and how it relates to the legacy pie-player-components item player — what it is, what changes, and why moving to it should be a smooth, low-risk update.

<pie-player> → <pie-item-player>

Part A · Item player

The big picture

Legacy pie-player-components

  • Stencil 3 custom-element library
  • Two separate elements: <pie-player> (delivery) and <pie-author> (authoring)
  • Item rendering solved; section, tools, and accommodations left to each product
  • Bundles loaded via IIFE script injection and window.pie globals

New @pie-players/pie-item-player

  • Svelte 5 + Vite, ESM-first custom element
  • One <pie-item-player> built on shared loaders / controllers / renderer
  • Drop-in for delivery; same job, modernized internals
  • Typed contracts, explicit controllers, provider-agnostic observability
What this means for your app: same role — render a PIE item — with safer markup, more loading options, versioned elements, resilient resource loading, and pluggable observability, behind a cleaner API.

Tidbit: delivery and authoring now live in the same element via a mode attribute, so there's no separate <pie-author> to wire up.

Part A · Item player

At a glance: legacy vs. new item player

Legacy <pie-player>

  • Framework: Stencil 3
  • Authoring: separate <pie-author>
  • Element tags: pp-*, version stripped
  • Loading: IIFE only
  • Markup: raw innerHTML (no sanitizer)
  • Session: inline in player
  • Resource loading: no in-item retry/tracking
  • Observability: New Relic, hard-wired
  • Theming: ad-hoc CSS

New <pie-item-player>

  • Framework: Svelte 5 + Vite
  • Authoring: same element, mode="author"
  • Element tags: versioned *--version-x-y-z
  • Loading: iife / esm / preloaded
  • Markup: DOMPurify by default
  • Session: dedicated ItemController
  • Resource loading: tracked + auto-retried
  • Observability: pluggable providers
  • Theming: @pie-players/pie-theme tokens
Part A · Change 1

Flexible loading strategies

One strategy attribute, three deployment models — vs. legacy IIFE-only script injection. This is the key to a low-risk migration.

iife

Compatibility path for your existing PIE bundle infrastructure and deployed content — the drop-in starting point.

esm

Modern dynamic import() from CDN / import maps. Smaller payloads, browser caching, view variants.

preloaded

Elements registered before render. Zero runtime fetch — offline-like, CI-built sets, test harnesses.

Benefit: start on iife to match your current bundles today, then opt into ESM / preloaded performance later — without changing the consumer API.
Part A · Change 2

Version coexistence by design

Legacy global, version-stripped

  • Tags rewritten to pp-<package>
  • Version stripped at controller lookup
  • Registered globally on window.PIE_REGISTRY
  • One version of an element per page

New versioned tags

  • makeUniqueTags() → multiple-choice--version-9-9-1
  • Version encoded into the tag & model IDs
  • Safe, guarded custom-element registration
  • Multiple element versions coexist at runtime
Benefit: authored content pins exact element versions; no global collisions when a page mixes versions (essential once a section composes many items).
Part A · Change 3

Security by default

Legacy

  • Item markup injected via raw innerHTML
  • No built-in sanitizer
  • XSS posture left to the host

New

  • DOMPurify on by default
  • Preserves the PIE contract: pie-* tags, id, model-*, data-*, aria-*
  • Opt out with trust-markup; customize via sanitizeMarkup
Benefit: XSS defense out of the box without breaking the versioned-element contract.
Part A · Change 4

Robust sessions + backend scoring

Legacy

  • Session logic inline in the player
  • Temporary event-blocking while setting models
  • Scoring: local provideScore() only

New

  • ItemController normalizes & de-dupes session updates
  • Metadata-only updates don't wipe responses
  • Local provideScore() and backend score(), autosave, save
Benefit: cleaner, explicit session ownership and an optional server-backed scoring / persistence path.
Part A · Change 5

Resilient resource loading

Items pull in images, audio, video, and stylesheets. The new player watches those loads and retries failures automatically — something the legacy player doesn't do for in-item resources.

Legacy

  • No tracking of item-embedded assets
  • No retry when an image / audio / video fails
  • Optional bundle fetch retry only, and only if re-fetch-bundle is set

New ResourceMonitor

  • Watches the item container for media & stylesheet loads
  • Automatic retry with exponential backoff (defaults: 3 tries, capped delay)
  • IIFE element bundles also retry while a bundle is still building
  • Tunable via loader-config: maxResourceRetries, resourceRetryDelay, iifeBundleRetry
Benefit: transient CDN / network hiccups self-heal instead of leaving a broken image or silent failure in front of a student — retries run even when telemetry is turned off.
Part A · Change 6

Observability you can plug in

Telemetry is provider-agnostic: the player emits structured signals; you decide where they go.

Legacy

  • New Relic hard-wired into the loader
  • Generic loader actions (e.g. api_loadScript_*)
  • No clean way to disable, redirect, or fan out

New InstrumentationProvider

  • Swap providers: New Relic, console, debug panel, or a composite fan-out
  • Explicitly disable with instrumentationProvider: null
  • Structured, named signals: pie-resource-load, pie-resource-retry, pie-iife-bundle-retry*, typed player-error
  • DOM lifecycle events (load-complete, session-changed, bundle-retry-status) for host wiring
Benefit: production telemetry through your existing backend, a local debug panel while developing, or both at once — with clearer, resource-level visibility than the legacy loader exposed.
Part A · Change 7

Theming, accessibility & packaging

Theming

Shared @pie-players/pie-theme token contract instead of ad-hoc per-product CSS.

Accessibility

Working base-heading-level & SR headings, WCAG 2.2 AA target, axe-tested demos.

Packaging

Clean dist ESM exports, auto-registration on import, typed public surface.

Benefit: consistent look, real accessibility posture, and a simpler, contract-checked consumption model.
Part B · Context (optional)

The bigger picture — if you ever need more than one item

You don't need this to adopt the item player. But the same project also covers full student-facing quiz / Knowledge Check (KC) screens, so you have a growth path.

Section player

Composes passages + multiple items into a complete KC screen (split-pane / vertical / tabbed layouts). A SectionController handles in-section navigation and aggregates item sessions. It embeds <pie-item-player> and passes your loading strategy through.

Assessment toolkit

Composable services for tools & accommodations a quiz needs: text-to-speech, highlighting, calculators, and QTI-style IEP/504 policy. Opt-in only — the item player works fine without it.

Takeaway: adopting the item player today doesn't commit you to anything else — but if your roadmap grows toward full quizzes, the section player and toolkit are already there.

What this means for you: you can adopt just the item player — the higher layers are optional. The whole @pie-players/* suite ships at one fixed (lockstep) version, so there's no compatibility matrix to manage.
Part C · What adoption looks like

Why this should be a smooth update

What stays the same

  • Same job: render a PIE item from config + session
  • Familiar attributes/props — most names carry over
  • strategy="iife" keeps your current bundle infrastructure
  • Legacy pie-item client behaviors preserved where externally consumed

What you gain

  • Secure-by-default markup; versioned, collision-free elements
  • Self-healing resource loads (auto-retry)
  • Pluggable observability for production and debugging
  • A typed, contract-checked API under active maintenance
In short: the switch is essentially swapping the tag and keeping iife loading — small, incremental, and low-risk — while quietly improving resilience and visibility. Details: docs/item-player/migration-from-pie-player-components.md.
Part C · Wrap up

Quick start & takeaway

<!-- drop-in delivery: keep your IIFE bundles -->
<pie-item-player
  strategy="iife"
  config='{ ... }'
  session='{"id":"s1","data":[]}'>
</pie-item-player>

// later, opt into modern loading
// strategy="esm" or "preloaded"

Takeaway

  • Same job as today, with flexible loading, versioned elements, and secure-by-default markup.
  • Quiet wins for your app: self-healing resource loads and pluggable observability.
  • The iife path keeps the switch small and low-risk; modernize internals on your own schedule.
  • Section player + toolkit are there if you grow toward full quizzes — but optional.
